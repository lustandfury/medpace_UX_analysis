Overview

The Three Tools — What They Share & Where They Differ

All three tools generate screens from text prompts and support iterative refinement. The difference is in where the output lives and who uses it next.

All Three
Generate from a text prompt Produce interactive prototypes Support iteration Work best with specific, realistic data
Conversational-native
Claude Design
  • Lives in claude.ai/design
  • Powered by Opus 4.7
  • Refine via conversation + inline edits
  • One-click handoff to Claude Code
  • Exports: URL, HTML, PPTX, Canva
  • Reads your design system automatically
Best for: Rapid exploration, stakeholder prototypes
Design-native
Figma Make
  • Lives in figma.com/make
  • 300 credits per prompt
  • Output copies to Figma canvas
  • Integrates with Figma design system
  • Best for: Screens that live in Figma
  • Real data, click-through interactions
Best for: Team annotation & Figma handoff
Code-native
Cursor + Claude Code
  • Lives in Cursor IDE
  • Flat subscription — no credit limits
  • Output is Tailwind/Shadcn HTML
  • Reads & writes actual code files
  • Can extract specs from prototypes
  • Generates Storybook stories
Best for: Dev-ready prototypes & docs
Workflow

How to Work With Each Tool

Each tool has a natural grain. Work with it rather than against it — use the right verb at each step.

Claude Design

Start

Describe in natural language

Be specific about user, context, data, and style

Refine

Continue the conversation

"Make the risk column more prominent"

Edit

Click directly on elements

Inline edits for text, sliders for spacing/colour

Branch

Add a second version with dark mode

Explore variations without losing the first

Handoff

One click to package for Claude Code

Or export to Canva / PPTX for stakeholders

Figma Make

Start

Describe the screen with specifics

Columns, data, style reference products

Refine

New prompt each time

Describe what to change — costs 300 credits per prompt

Copy

Click 'Copy to Figma'

Prototype lands on your canvas, ready for annotation

Annotate

Switch to Figma proper

Add comments, apply your design system, hand off to dev

Embed

Embed Make prototypes inside Figma files

Also FigJam and Slides

Cursor + Claude Code

Start

Open Cursor, create a file

Cmd+L to open chat, paste your prompt

Refine

Chat in the same panel

"The filter toggle isn't working — fix it"

@Attach

Type @filename to attach any file

PRD, screenshot — for context

Read

Ask Cursor to read a prototype

Extract design decisions (Evolved Spec workflow)

Ship

Output is Tailwind/Shadcn code

Kelly takes it straight to a PR — no retranslation

Principles

Best Practices for UX & UI Design

These apply across all three tools. The model is capable — your judgment is the constraint.

01
Specificity beats brevity

Name the user, the moment, the data. 'A CRA monitoring 6 oncology sites reviewing overdue visits' gets a better screen than 'a monitoring dashboard'.

02
Real data changes everything

Stakeholders give vague feedback on Lorem Ipsum. They give specific, actionable feedback on CDR-004 — Protocol Deviation, 38 days overdue.

03
Iterate, don't restart

First prompt establishes structure. Second refines style. Third locks in interactions. A 3-prompt sequence beats one exhaustive prompt every time.

04
Name the design language

Similar to Salesforce or Jira — dense, functional, not consumer. It closes the gap between what you imagine and what generates. Reference products are a superpower.

05
Your judgment is the product

AI generates options. You decide which are right. Your job shifts from pixel-pushing to evaluation and direction — a higher-order skill, not a lesser one.

06
Prototype to discover, not confirm

You can now afford to be wrong. Explore 5 directions in the time it used to take to build 1. The first direction is rarely the best one.
