Seven artifacts that chain together — each one feeding the next.

A markdown document defining the core problem, design priorities, edge cases, and success criteria. The foundation everything else builds on.

Before You Type Anything

Best first actions.

How you set up Figma Make before prompting shapes output quality more than any single word in your prompt.

Recommended path

If you have a design ready — attach it first

Click the + attachment icon and attach your Figma frame before writing a word. This is the biggest unlock in Figma Make. It gives Make visual grounding — it understands your actual components, spacing, and colour tokens rather than inventing them from scratch. The difference in output quality is significant.

Medpace example: Attach a frame from your clinical monitoring file. Make will apply your existing badge styles, nav structure, and colour tokens — instead of defaulting to generic SaaS blue.
Starting from scratch

Pick your mode intentionally — before prompting

The four buttons at the bottom of Make constrain how it generates output. Picking the wrong mode before prompting is the most common cause of layouts that fight you — Make optimises structure for that mode from the first generation, and changing modes later forces a full rebuild.

Rule: Set the mode before you paste your prompt. Changing mode mid-session means starting over — you lose whatever structure was working.
The Four Modes
Visualization
Data · charts · dashboards · KPI views
Medpace — use for
Patient enrollment tables, site performance dashboards, protocol deviation summaries, enrollment velocity charts
Website
Marketing pages · portals · landing pages
Medpace — use for
Sponsor-facing study portals, participant recruitment pages, study information pages
Component
Isolated UI elements — best for iteration
Medpace — use for
Status badge variants, patient record panels, filter toolbars, eligibility form sections, risk flag components
Mobile App
Constrained to mobile viewport
Medpace — use for
CRA pre-visit summary view, site coordinator quick-update screen, participant check-in flow
Use the LLM to set up

You don't have to figure out which mode or prompt structure to use alone. Describe what you're building to Claude first — the screen, the user, what they're trying to do — and ask it to help you write the Figma Make prompt. A short conversation catches gaps before Make does.

Step 01

Problem Framing

Do this before you touch a single frame.

Ask Make to generate a design brief first. This forces clarity on what you're actually solving and gives you a document to attach to every subsequent session. Once attached, Make always knows the "why" behind every decision without you re-explaining it.

Figma Make Prompt
Generate a design brief as a markdown document for the Medpace Patient Enrollment Management Platform. Context: CRAs (Clinical Research Associates) manage 10 to 40 patients across 3 to 8 active trial sites at once. Current workflow: spreadsheets and email. The platform needs to give a CRA one place to see what needs attention, what's overdue, and whether they're ready for the next monitoring visit. The reference trial is ONCO-2024-003, with 12 patients across Cedars-Sinai Medical Center (LA), Mayo Clinic Rochester MN, and Johns Hopkins Hospital Baltimore MD. The core tension: CRAs need dense clinical data to do their job accurately, but they also need to triage fast — pre-visit the morning before travel, on-site with limited time and focus. The design must support both modes without switching views. For the brief, include: - The core user problem stated in one honest sentence — not a marketing sentence, a real one - 3 genuine design tensions we'll have to make calls on (e.g. "displaying complete patient data vs. enabling fast pre-visit triage") - 5 specific edge cases: a protocol deviation requiring PI escalation, a site where all patients are on treatment hold, a visit window expiring on a day the site is closed, a withdrawn patient still under an open query, a study amendment that changes required visit counts mid-trial - What "done for the day" actually looks like for a CRA — not just task completion, but the professional confidence state we're designing toward (she should be able to close the laptop and trust the work is in order) - 3 products this should never feel like, and why each one is a cautionary tale for us specifically in the clinical context - 4 success metrics that are actually measurable, not vanity metrics Format with clear markdown headers. This document gets attached to every subsequent Make session.
Generate a design brief as a markdown document for an app called [product name] — [what it does and who it's for]. Include: - The core user problem in one sentence - 3 competing design priorities - 5 edge cases the design must account for - What "done" looks like for the user - 3 things this product should never feel like - Success metrics we could track Format with clear headers. I will attach this to all future Make sessions.
Why this works

The brief becomes a reusable artifact. Attach it to every subsequent Make session using the + button in the prompt bar. Make uses it as silent context — keeping outputs aligned to the same problem without you re-explaining each time.

→Save the output as a .md file alongside your Figma file throughout the project.
→Be specific about what it shouldn't feel like — that constraint is often more useful than describing what it should be.
→Not sure what your problem actually is? Ask Claude: "I want to design [X]. Help me find the real design problem before I write a brief." Use it to think out loud before committing to a direction.

Step 02

User Personas

Give Make a human to design for.

Generate structured persona documents as markdown. These aren't just for your team — they become context you attach to Make so every generated component reflects real user needs, not generic patterns. The more specific the persona, the more distinctive the output.

Figma Make Prompt
Generate 3 user personas as a markdown document for the Medpace Patient Enrollment Management Platform. I want these to feel like people I could debrief with after a site visit, not composite sketches. Here's the starting direction for each: Persona 1: A CRA, approximately 28–36, travels three days a week to trial sites. Currently managing 6 active sites across two protocols. Her primary systems are Medidata Rave for data entry and Excel plus email for visit prep and follow-up. Master's degree in life sciences. Gets anxious the night before site visits when she hasn't had time to prep properly — the anxiety isn't about the site, it's about not knowing what she's walking into. Persona 2: A Site Coordinator — the person at the academic medical center who receives CRA visits, not the person conducting them. Late 30s. Has been at the same institution for 8 years. Manages 25–40 patients across two active trials simultaneously, using the EHR, an EDC, and her own Excel tracking spreadsheet at the same time. Deeply skeptical of new software — not because she's averse to technology, but because every system change creates compliance risk until the SOPs are updated and the team is retrained. She has been burned before. Persona 3: A Clinical Ops Manager, 40s, oversight role. Reviews monitoring reports, responsible for escalation decisions, rarely looks at individual patient data — she needs site-level risk at a glance. Describes herself as "a dashboard person." Has previously tried to get useful summary reporting out of Medidata Rave and found it practically unusable for her purposes. Approves the platform budget but isn't the daily user. For each persona, give me: - Their name, age, location, and job title exactly as it would appear on their business card - The 3 systems they use daily and what specifically frustrates them about each one - The thought in their head at 7am on the morning of a site visit (or Monday 8am for the Ops Manager) - A verbatim quote — something they'd actually say in a handoff meeting, unfiltered - The moment in their week when this platform would give them the most relief - Their gut reaction to new internal tools: compliance-cautious / pragmatic / cautiously enthusiastic - One specific thing in the UI that would make them immediately distrust the data
Generate 3 detailed user personas as a markdown document for [product name], [one sentence description]. For each persona include: - Name, age, location, job title - Tools they use and what they hate about them - Their biggest daily frustration - What they wish existed - A quote they'd actually say out loud - How they define a successful workday - Relationship with technology: comfortable / reluctant / power-user - One thing they'd notice immediately if the design got it wrong Make them feel like real people, not archetypes.
Less obvious use

After generating personas, ask Make to pick the most underserved persona and explain what a design optimised specifically for them would prioritise differently. This surfaces directions you wouldn't reach any other way.

→Ask for conflict — request personas with competing needs. Design tension found early is easier to resolve than tension found in review.
→Follow up with: "Generate an interactive HTML version of these personas" — shareable with anyone, no Figma access needed.

Step 03

Brand Brief

Stop Make from going off-brand before it starts.

Generate a design token and style reference document. Attach this to every Make session. It's the single most efficient way to get consistently on-brand output — far more effective than repeating style instructions in every prompt.

Figma Make Prompt
Create a brand and design system reference document for the Medpace Patient Enrollment Management Platform. I have a clear point of view on the brand direction, which I want you to develop into a full system: The feeling: this platform should feel like a reliable professional instrument. Not a dashboard, not a portal — a clinical instrument a CRA trusts because it's never wrong. References for the feeling I'm after: aircraft cockpit indicator clarity (critical information is always visible, hierarchy is non-negotiable), clinical lab report authority (structured, precise, no decoration that doesn't carry meaning), Bloomberg Terminal density (high information per square centimetre without visual chaos). What it absolutely must not feel like: - Salesforce default configuration: too much chrome, navigation hierarchy that prioritises navigation over content, "sky blue" that reads as generic enterprise - Apple Health or any consumer health app: wrong register entirely for a regulated clinical environment — the warmth and encouragement that works for fitness tracking is inappropriate when a CRA is reviewing a protocol deviation - Legacy EDC systems like Medidata Rave: dated visual design that creates friction, slows clinical judgment, and signals that the tool wasn't designed with the user in mind Colour direction is defined. Work with it: - Primary surface: white (#FFFFFF) - Action / status: clinical teal (#028090) - Navigation / headers: deep navy (#0D1B2A) - Risk states must be red and amber, but color-blind safe — deuteranopia and protanopia both occur in the user base. Color-alone communication is not acceptable. Every status must pair color with an icon or label. Please develop: - Full primary / secondary / accent / semantic palette. Hex values and explicit usage rules for each. Semantic colors must include: Critical (with background tint and foreground color passing WCAG AA on white), Warning, Success, Info — all four. - Typography: a serif for page-level titles that communicates authority, not warmth (rule out anything typically used for lifestyle brands); a clean sans-serif for data labels and UI copy that remains legible at 12–13px in dense tables; a monospace for patient IDs, timestamps, and protocol codes. Full scale from 11px to 32px with usage notes at each size. - Spacing scale, 4px base unit. Tokens space-1 through space-10. - Table-specific tokens: row height (compact / default / spacious), cell padding, header background, border weight. This is a data-dense tool — table tokens matter as much as component tokens. - Tone of voice: 4 example UI strings written in the correct clinical register — precise, direct, assumes expertise. Then: 5 words that must never appear anywhere in this product's copy. Format as markdown with clear section headers. This gets attached to every Make session from this point.
Create a brand and design token reference document for [product name]. The brand should feel [3 adjectives]. Not [what to avoid]. Include: - Color palette with hex values and usage rules - Typography: heading, body, mono fonts with size scale - Spacing system (4px base unit) - Border radius and shadow definitions - Tone of voice: 3 adjectives, 3 anti-adjectives, writing example - 2 brands as anti-inspiration - Component personality notes (buttons, inputs, empty states) Format as markdown. I will attach this to future Make sessions.
Pro move

Ask Make to generate a visual HTML style guide instead of markdown — a living reference page that renders the actual colors, type scale, and component styles in the browser. Far more useful to share with a team or client than a text document.

Step 04

Microcopy

Words are part of the design. Generate them before you build.

Generate all UI copy upfront — buttons, errors, empty states, tooltips, onboarding text. When you attach this to your component prompts, Make uses real copy instead of placeholder text. This single change transforms what prototype reviews feel like.

Figma Make Prompt
Generate a complete UI microcopy document for the Medpace Patient Enrollment Management Platform. Tone guidance — this is critical: Write for a regulated clinical environment. The CRA using this tool is a trained professional operating under 21 CFR Part 11 compliance. Write like a senior colleague who respects their expertise. Precise, direct, zero ambiguity. No "simply", "easily", "just", or "quickly" — these words are filler and they're inappropriate in a compliance context. No exclamation marks anywhere. No success celebrations. Error messages must tell the user what to do next, not just that something went wrong. Confirmations must confirm exactly what was recorded — a CRA may cite these in a monitoring report. Screens to cover: 1. Patient list — empty state when filters return zero results This is not a new study — data exists, the user has over-filtered. Do not suggest onboarding actions. Acknowledge the filter state and give a clear exit. Do not show a bare empty table. 2. Patient list — newly created study with no enrolled patients yet Genuine zero state. Matter-of-fact. The next action to enroll patients is not in this tool — do not suggest one. 3. Flag review confirmation — after logging a protocol deviation This is a compliance action. The confirmation must be specific about what was recorded. Write it as if the CRA needs to be able to describe what they logged in a monitoring report. 4. Contact attempt logging — field labels and placeholder text Fields: Patient ID (read-only), Contact Date, Contact Method (dropdown), Outcome (dropdown — options: Reached / No Answer / Left Message / Incorrect Contact), Notes (optional). Clinical-precise, not casual. 5. Error states: session timeout during a flag submission (preserve all entered data, prompt re-authentication, return to review step), EDC sync failure (flag queued locally, inform CRA it will submit when connection restores), read-only access attempt (CRA attempting to submit on a site they don't have write access to — be specific about why) 6. Visit window status badge labels Exact label text for each state: Upcoming, Due Soon, Overdue, Missed, Completed. One to two words maximum. Must be unambiguous in a table cell at small size. 7. Navigation labels: 4 top-level nav items. "Dashboard" is technically accurate but generic — what label better reflects a CRA's mental model of her primary view? Output as markdown with each screen as a section header.
Generate a complete UI microcopy document for [product name]'s [screen or flow]. Cover: - Page titles and headings - Primary and secondary CTA labels - Empty states: headline + 1 sentence + CTA for each major empty state - Error messages: network, validation, limits - Success and confirmation copy - Tooltip text for key features - Onboarding text and coach marks - Input placeholder text Tone: [describe voice]. Avoid: [words or patterns to exclude]. Output as structured markdown by UI section.
Why this matters

Placeholder copy causes stakeholders to react to the layout instead of the experience. Real copy shifts the entire conversation in review. Generate it first, attach it when building — the difference in feedback quality is immediate.

Step 05

User Flow

Map the journey before you build any screen.

Ask Make to generate a clickable HTML user flow — an interactive diagram of screens and decision points. Faster to produce than a Figma flow, works in any browser, and is far easier for non-designers to review asynchronously.

Figma Make Prompt
Build an interactive HTML user flow diagram for the Medpace Patient Enrollment Platform's "Flag Patient for Review" flow. Here's exactly what happens: Entry point: CRA is reviewing the patient list, notices an amber "At Risk" badge on a row. She hovers the row — a three-dot action menu appears on the right. She clicks "Flag for Review." Step 1 — Flag type selection (required) Modal opens. Single required selection before proceeding: Protocol Deviation / Missed Visit Window / Unanswered Contact Attempt / Lab Result Pending Review / Other. Selecting "Other" reveals a required note field. Cannot proceed without selecting a flag type. Step 2 — Details (two paths based on Step 1 selection) Path A — Protocol Deviation selected: required fields are Deviation Date (date picker), Protocol Section Reference (free text), Brief Description (required, max 500 characters). Optional: attach supporting document. Path B — All other flag types: required field is Brief Description (required, max 500 characters). Optional: attach supporting document. Step 3 — Assignment Dropdown: Assign to Self / Assign to Site Coordinator / Escalate to PI / Escalate to Sponsor. If "Escalate to PI" or "Escalate to Sponsor" is selected, a required "Reason for escalation" text field appears. Cannot proceed without completing it. Step 4 — Review & Submit Summary of all entered information. Non-editable except via Back button. Primary CTA: "Submit flag." Visible warning annotation: "This action will be logged in the audit trail and cannot be deleted." Success state Flag submitted confirmation displaying: generated flag ID (format FLAG-YYYYMMDD-XXX), patient ID, flag type, and assigned to. Two CTAs: "Return to patient list" or "View flag in patient record." Error states to show: - Session expired during flag submission: preserve all entered data, prompt re-authentication, return to Review & Submit step with data intact - EDC sync failure: flag queued locally, banner informs CRA it will submit automatically when connection restores Style: dark background diagram. Three distinct path colors — happy path, escalation branch, error recovery. Label every arrow with its trigger action. Rounded rectangle nodes. Include the audit trail warning as a visible annotation on the Submit node. Every node must show the step name and a one-line description inside.
Build an interactive HTML user flow diagram for [product name]'s [flow name]. Flow starts: [trigger action] Flow ends: [success state] Steps: [List key steps and decision points] Show: happy path, any skip paths, [N] error states. Style: dark background, styled nodes, directional arrows. Shareable with non-technical stakeholders.
Unexpected use case

Generate the flow before you have any opinion on it — just describe the user goal. Make will often surface steps you forgot or decision points you hadn't considered. Use it as a thinking tool first, documentation tool second.

Step 06

Build the Component

Now you're ready. And you have context behind you.

This is the prompt most people start with. Starting here without steps 1–5 means Make guesses at the problem, the user, the brand, and the copy. With those artifacts attached via the + button, this becomes a fundamentally different request. The example below is a Medpace clinical monitoring screen — the principle applies to any screen you're building.

Attach: brief · personas · brand doc · microcopy
Build Medpace's patient enrollment management screen for ONCO-2024-003 as a React component. I'm attaching the design brief, CRA persona doc, and brand tokens — use them. Don't guess at brand decisions; if the brand doc specifies something, follow it exactly. Layout I want to see: Sticky header strip: - Study name "ONCO-2024-003" left-aligned + enrollment progress bar showing 8/15 (53%) right of it - Below the title: four stat cards in a row — Total Screened (12) · Enrolled (8) · Withdrawn (1) · Completed (1) - These stats are read-only — no click actions Patient table below the header: Columns in this order: Patient ID · Site · Status · Enrolled Date · Last Visit · Days Since Visit · Risk Flag · Next Action 12 rows of realistic clinical data across three sites: Cedars-Sinai Medical Center (LA) · Mayo Clinic Rochester MN · Johns Hopkins Hospital MD Status mix across rows: Screened ×2, Enrolled ×5, On Treatment ×3, Withdrawn ×1, Completed ×1 Risk flags: 1 Protocol Deviation (red badge), 2 At Risk (amber badge), rest empty Days Since Visit: colour-code red if >30 days, amber 15–30, green 0–14 — these are real clinical thresholds Filter bar above table: Status dropdown · Site dropdown · "Show flagged only" toggle with a live badge count showing how many flagged rows exist. Toggling it filters the table immediately. Each row: three-dot action menu (rightmost). Specific requirements: - Risk Flag column must be the most visually dominant element in each row — a CRA's eye should land there first - Hover on each row: reveal the Next Action button — don't show it when not hovering - Empty state when no patients match active filters: "No patients match your current filters." + "Clear all filters" link below it. Do not show a bare empty table. - The full table must be keyboard navigable — logical tab order, Enter triggers the row's Next Action Style: Internal enterprise clinical monitoring tool. White table on light grey background. Teal/navy header strip. High information density. Closer to Salesforce Health Cloud than any consumer app. Not playful. Not rounded. Tech: React functional component, Tailwind for styling, no other dependencies. Single file.
Build [product name]'s [screen or component name] as a [React component / HTML page]. Using the attached design brief, personas, brand doc, and microcopy as your guide: Layout: [Describe key sections and hierarchy] Requirements: - Match brand tokens from the attached doc exactly - Use copy from the microcopy doc — no placeholder text - Optimise for [primary persona + key behaviour] - Hover states and transitions throughout Tech: [React + Tailwind / HTML + CSS]. Single file.
The compounding effect

By attaching 4 documents, you're giving Make the equivalent of a design system + user research + creative brief + content strategy in one session. The output difference compared to a blank prompt is categorical, not incremental.

→Use Website mode for full pages, Component mode for isolated UI elements.
→One screen at a time. Scope creep in the prompt produces scope creep in the output.
→Use Claude as a prompt co-author. Describe the screen in plain language and ask: "Turn this into a precise Figma Make prompt." A 3–4 message conversation usually produces a better first prompt than writing one from scratch alone.

Step 07

Iterate & Refine

The first output is a skeleton. These prompts shape it.

Follow-up prompts are where most of the craft happens. These six patterns each target a different failure mode in generated output. Use them individually — one at a time — and always describe what you want rather than what went wrong.

High-Leverage Follow-Up Prompts
// Hierarchy — risk flags aren't landing first "The risk flag column exists but my eye doesn't land there first — it lands on the Patient ID column, which is the least actionable thing in the row. A CRA opening this Monday morning needs to see critical flags immediately. Increase the visual weight of the Risk Flag column: the red Protocol Deviation badge should be the most visually prominent element in the row. Reduce the weight of Patient ID — smaller type, lighter color — it's reference data, not decision data. Don't move columns. Adjust visual hierarchy only." // Accessibility — status badges are color-only "This table communicates patient status (red Protocol Deviation, amber At Risk) using color alone. This fails WCAG 2.1 AA for color-blind users — deuteranopia affects approximately 8% of males. For every status badge, add a symbol prefix: ▲ for Protocol Deviation, ● for At Risk, leave clean otherwise. Also: tab through the entire table — every row must be keyboard-navigable with Enter triggering the primary action, and the three-dot menu must be reachable without a mouse. Fix the visit window status pills too — check them for contrast." // Responsive — tablet use case for on-site CRA review "A CRA occasionally checks the patient list on a tablet while physically at a site. At 768px: hide Last Visit Date and Days Since Visit columns — replace with a single 'Last Contact' column showing the most recent of either. Collapse the three-dot action menu to a single 'Review' button visible on row tap. The filter bar should stack to two rows. The header stat cards should scroll horizontally rather than wrap. The table itself must not require horizontal scroll at 768px." // Dashboard header — make it visit-aware, not just a label "The current header shows the study name and enrollment count, but a CRA uses this as her first view before a monitoring visit — she needs it to tell her something actionable. Replace the static header with a context-aware status line: if a monitoring visit is scheduled in the next 5 days, show 'Monitoring visit: [Site Name] in [N] days — [X] outstanding items to resolve.' If there are critical flags, surface the count. If everything is clear, show 'All sites ready — no critical items.' Write 4 different versions of this line for different states and implement the logic to show the right one." // Data states — table only handles the loaded state "The patient table has no loading state, no error state, and no empty state. Add all three: (1) Loading — skeleton rows with the same column dimensions as the real rows, animated shimmer, show 8 of them (matching a typical CRA patient load). (2) Error — EDC sync failed; show an inline banner at the top of the table with the timestamp of the last successful sync and a 'Retry sync' button. Do not show a bare empty table. (3) Empty (filtered) — use exactly this copy: 'No patients match your current filters.' with a 'Clear filters' link. No illustrations, no friendly empty-state icons — this is a clinical tool." // Persona check — run it against the site coordinator "Read the site coordinator persona in the attached personas doc — the one who has been at the same hospital for 8 years, manages 25–40 patients, and is deeply skeptical of new software because every system change creates compliance risk until SOPs are updated. She's been asked to use this dashboard for the first time. Based on what you know about her, tell me: what are the 3 things most likely to make her distrust the data or feel like this tool creates more compliance risk than it eliminates? Then fix those 3 things without breaking the CRA-optimized view."
// Hierarchy Fix "The most important action is [action]. Redesign visual hierarchy so it draws the eye first." // Accessibility "Run an accessibility review. Fix contrast, focus states, missing labels, and form field associations." // Responsive "Update for mobile: [describe breakpoint behaviour]." // Motion "Add micro-interactions: [hover states, transitions, load animations]." // Data States "Add: loading skeleton, empty state with copy '[microcopy]', error state. All on-brand." // Persona Check "Review against the [persona name] persona in the attached doc. What frustrates them? Fix the top 3 issues."
The Persona Check

The last prompt is the most underused. Asking Make to critique its own output through the eyes of a specific user — with the persona doc attached — surfaces issues that no amount of visual polish will fix. Run it before you call any screen done.

→One change at a time. Stacking instructions leads to unpredictable changes across the component.
→If a change goes wrong, describe what you wanted rather than what you got.
→Save versions you like before making significant changes — Make doesn't have granular undo.
→Stuck on what to try next? Explain the problem to Claude: "Here's what Make generated, here's what I actually need — write me a better follow-up prompt." Let it draft the iteration.

Common Mistakes

What to avoid in first prompts.

These patterns produce outputs you'll spend more time fighting than refining.

❌ Too vague
"Make a clinical monitoring dashboard"
No user, no data, no layout constraints. Make produces a generic chart grid with placeholder headers. You'll spend the next five prompts trying to recover structure instead of moving forward.
✓ Specific — name the user, the data, the column
"Design a CRA monitoring screen with an 8-column patient table including Days Since Last Visit (colour-coded red/amber/green), a sticky header showing study ONCO-2024-003 and 8/15 enrollment progress, and a 'Show flagged only' filter toggle."
Names the user (CRA), the key differentiating column, a real interaction, a specific study ID, and a concrete UI element. Make can generate something immediately useful.
❌ Too much at once
"Build a clinical trial management platform with a patient table, site dashboard, sponsor reporting module, protocol deviation tracker, and admin settings."
Make will generate something that looks like an app but works like a wireframe of nothing in particular. Scope one screen. Get it right. Move to the next screen in a separate session.
✓ Scoped — one screen, one user, one moment
"Design just the patient enrollment table — the main monitoring view a CRA opens on Monday mornings to review all active patients across their 3 assigned sites. That's the whole scope."
One screen, one user, one workflow moment. Make generates the full screen well. The site dashboard comes in its own session once this one is right.
❌ No style direction
"Build a participant eligibility screening form."
Without a visual reference, Make defaults to a generic consumer-app style — rounded, colourful, friendly. Clinical tools need to feel structured and authoritative. The aesthetic mismatch will take 3–4 prompts to correct.
✓ Toned — add a reference and a "not this"
"Build a participant eligibility screening form. Style: internal clinical research tool — white background, structured section headers, accessible typography. Similar to REDCap but significantly better designed. Not a consumer app, not playful."
"Similar to REDCap" gives Make a concrete reference it understands. "Not a consumer app" explicitly rules out the default aesthetic. The first output needs far less correction.
Before you paste into Make

Run your prompt by Claude first: "I'm about to use this Figma Make prompt — what's missing or too vague?" It catches the same gaps Make will struggle with. This is especially useful for your first prompt on a new screen type, and for writing Skills you plan to reuse — you don't have to draft either from scratch.

Pro Tip

"Treat the first generation as a skeleton, not a final result. Get the structure right first, then iterate on style."

What this means in practice: If the layout is wrong after two targeted prompts, start over with a tighter brief — don't try to fix structure through style prompts. Once the skeleton is right (correct columns, correct hierarchy, correct sections), follow-ups like "make the nav sticky" or "replace the chart with a bar chart" are fast and reliable. A well-structured clinical data table with imperfect styling is three prompts from done. A poorly-structured screen with beautiful styling is a restart.
