Overview

What this kit produces

Seven artifacts that chain together — each one feeding the next.

1

Problem Brief

A markdown document defining the core problem, design priorities, edge cases, and success criteria. The foundation everything else builds on.

2

User Personas

Structured persona docs with goals, frustrations, behaviours, and one critical design implication each. Attach to Make so it designs for real humans, not assumptions.

3

Brand Brief

A living design token reference — colors, type scale, spacing, tone of voice, and what the brand should never look like. The fastest path to consistently on-brand output.

4

Microcopy Document

All UI text upfront: buttons, errors, empty states, onboarding, tooltips. Attach this so Make uses real copy instead of placeholders — transforms what prototype reviews feel like.

5

Interactive User Flow

A clickable HTML flow diagram with happy paths, skip paths, and error states. Faster to build and share than a Figma flow, and works in any browser.

6

Component Build

The actual Make generation prompt — written to reference all prior artifacts. The output quality difference versus a blank prompt is categorical, not incremental.

7

Iteration Prompts

Six high-leverage follow-up prompts for hierarchy, accessibility, responsiveness, data states, and the Persona Check — the most underused review in the kit.

● The example project throughout is Luma — a project management app for freelancers managing multiple clients. Each step has an Example tab (Luma) and a Template tab (fill-in-the-blank).
Step 01

Problem Framing

Do this before you touch a single frame.

Ask Make to generate a design brief first. This forces clarity on what you're actually solving and gives you a document to attach to every subsequent session. Once attached, Make always knows the "why" behind every decision without you re-explaining it.

Figma Make Prompt
Generate a design brief as a markdown document for an app called Luma. Context: Luma is a project management tool for freelancers — specifically people who run a solo practice and juggle 4 to 8 active client projects at once. Not agencies. Not teams. Individual practitioners who are the designer, the account manager, and the invoice chaser all at the same time. The core tension: these users are incredibly time-poor, but their work is complex enough that a simple to-do list isn't enough. Tools like Notion are too flexible (too much setup). Tools like Asana feel like they were built for teams and make solo use feel lonely and bureaucratic. We're trying to land somewhere between those two failure modes. For the brief, include: - The core user problem stated in one honest sentence — not a marketing sentence, a real one - 3 genuine design tensions we'll have to make calls on throughout the project (e.g. "showing enough detail vs. keeping the overview scannable") - 5 specific edge cases: think about someone mid-project with a client who ghosts them, someone who works across 3 time zones, someone who invoices in a currency that isn't USD - What "I'm done for the day" actually looks like for this user — not just task completion, but the emotional state we're designing toward - 3 products this should never feel like, and why each one is a cautionary tale for us specifically - 4 success metrics that are actually measurable, not vanity metrics Format with clear markdown headers. This document gets attached to every subsequent Make session.
Generate a design brief as a markdown document for an app called [product name] — [what it does and who it's for]. Include: - The core user problem in one sentence - 3 competing design priorities - 5 edge cases the design must account for - What "done" looks like for the user - 3 things this product should never feel like - Success metrics we could track Format with clear headers. I will attach this to all future Make sessions.
Why this works

The brief becomes a reusable artifact. Attach it to every subsequent Make session using the + button in the prompt bar. Make uses it as silent context — keeping outputs aligned to the same problem without you re-explaining each time.

→Save the output as a .md file alongside your Figma file throughout the project.
→Be specific about what it shouldn't feel like — that constraint is often more useful than describing what it should be.

Step 02

User Personas

Give Make a human to design for.

Generate structured persona documents as markdown. These aren't just for your team — they become context you attach to Make so every generated component reflects real user needs, not generic patterns. The more specific the persona, the more distinctive the output.

Figma Make Prompt
Generate 3 user personas as a markdown document for Luma. I want these to feel like people I could call on the phone, not composite sketches. Here's the starting direction for each: Persona 1: A UX designer, probably 30–38, based in a mid-size city (not SF or NYC). Left an agency 2 years ago to go solo. Has 5 active clients right now and is drowning in Slack messages from all of them. Uses Notion but has 4 different project templates and none of them are right. Checks their bank account more than they'd like to admit. Persona 2: A copywriter or brand strategist, 40s, has been freelance for a decade and is actually good at it — but hates the admin side with a passion. Uses a spreadsheet to track projects, a separate spreadsheet for invoices, and a notes app for everything else. Not particularly technical. Would describe themselves as "organised in my own chaotic way." Persona 3: A developer who freelances on the side while holding a full-time job. Juggles 2–3 side projects at a time. Very comfortable with tools and has tried everything — Jira, Linear, Basecamp, ClickUp — and hated all of them for being either too heavy or too simple. Has strong opinions. Probably has a custom Notion setup they built themselves. For each persona, give me: - Their name, age, city, and what they'd put as their job title on a LinkedIn profile - The 3 tools they use most for managing work and what specifically annoys them about each - The thought they have most often while working at 4pm on a Thursday - A verbatim quote — something they'd say in a Zoom call, casual, unguarded - The moment in their week when Luma would give them the most relief - Their gut reaction when they see a new app: excited, skeptical, or "I'll try it once" - One thing in the UI that would make them close the tab immediately
Generate 3 detailed user personas as a markdown document for [product name], [one sentence description]. For each persona include: - Name, age, location, job title - Tools they use and what they hate about them - Their biggest daily frustration - What they wish existed - A quote they'd actually say out loud - How they define a successful workday - Relationship with technology: comfortable / reluctant / power-user - One thing they'd notice immediately if the design got it wrong Make them feel like real people, not archetypes.
Less obvious use

After generating personas, ask Make to pick the most underserved persona and explain what a design optimised specifically for them would prioritise differently. This surfaces directions you wouldn't reach any other way.

→Ask for conflict — request personas with competing needs. Design tension found early is easier to resolve than tension found in review.
→Follow up with: "Generate an interactive HTML version of these personas" — shareable with anyone, no Figma access needed.

Step 03

Brand Brief

Stop Make from going off-brand before it starts.

Generate a design token and style reference document. Attach this to every Make session. It's the single most efficient way to get consistently on-brand output — far more effective than repeating style instructions in every prompt.

Figma Make Prompt
Create a brand and design system reference document for Luma. I have a strong point of view on the brand direction, which I want you to develop into a full system: The feeling: opening Luma should feel like walking into a very well-organised workspace — not a sterile office, not a cosy coffee shop. Think: a freelancer's studio apartment where everything has a place and the light is good. Calm. Focused. A little bit considered. The kind of tool that respects that the person using it is good at their job. What it absolutely must not feel like: - Asana: too cheerful, too much colour, designed for team cheerleading rather than solo focus - Notion: too neutral and beige — feels like a blank sheet of paper, not a tool - Harvest: looks like it was designed in 2014 and nobody updated it Colour direction I'm leaning toward: a near-black or very dark navy for the primary surface (not pure black), with a warm off-white for text, and a single accent colour that reads as "smart" — something like a muted amber or a dusty teal. Not green (too "done"), not blue (too corporate), not purple (too startup-y). Please develop: - A full primary / secondary / accent palette. Give me hex values and a clear rule for when each is used. Include semantic colours for success, warning, and error. - Typography: I want a serif for display headings (something with character, not Playfair — too overdone), a clean sans for body and UI, and a mono for data and times. Include the full size scale from 11px to 48px with usage notes. - Spacing scale based on 4px. Show me tokens from space-1 through space-12. - Elevation: 3 shadow levels — subtle (card), medium (dropdown), prominent (modal) - Tone of voice: write me 5 example UI strings that demonstrate the voice. Then tell me the 5 words that should never appear in this product's copy. Format as markdown with clear section headers. This gets attached to every Make session from this point.
Create a brand and design token reference document for [product name]. The brand should feel [3 adjectives]. Not [what to avoid]. Include: - Color palette with hex values and usage rules - Typography: heading, body, mono fonts with size scale - Spacing system (4px base unit) - Border radius and shadow definitions - Tone of voice: 3 adjectives, 3 anti-adjectives, writing example - 2 brands as anti-inspiration - Component personality notes (buttons, inputs, empty states) Format as markdown. I will attach this to future Make sessions.
Pro move

Ask Make to generate a visual HTML style guide instead of markdown — a living reference page that renders the actual colors, type scale, and component styles in the browser. Far more useful to share with a team or client than a text document.

Step 04

Microcopy

Words are part of the design. Generate them before you build.

Generate all UI copy upfront — buttons, errors, empty states, tooltips, onboarding text. When you attach this to your component prompts, Make uses real copy instead of placeholder text. This single change transforms what prototype reviews feel like.

Figma Make Prompt
Generate a complete UI microcopy document for Luma's project dashboard, new project creation flow, and onboarding sequence. Tone guidance — this is important: Write like a smart friend who also happens to be a good project manager. Direct. A little dry. Never chirpy. No exclamation marks anywhere in the product. No words like "simply", "easily", "just", "quick", or "seamlessly" — these are filler words that make copy feel dishonest. Don't say "Oops!" for errors. Don't say "Woohoo!" for success. The user is a professional; treat them like one. Screens to cover: 1. Dashboard — empty state (user has no projects yet) Write a headline (max 6 words), one sentence of context, and a CTA button label. The tone should acknowledge the blankness without being apologetic about it. 2. Project list — partial empty state (user has projects but none match a filter they've applied) Different from above — the user isn't new, they just filtered to nothing. Don't repeat the onboarding energy. 3. Invoices tab — empty state This one can be slightly warmer — an empty invoice list is actually a good reminder, not a failure state. 4. New project form — all placeholder text Fields: Project name, Client name, Deadline (optional), Notes, Hourly rate. Each placeholder should feel like a helpful hint, not filler. 5. Error states: network timeout, duplicate project name, invoice amount over $99,999 (validation) 6. Success messages: project created, invoice sent, timer stopped 7. Navigation labels: 5 top-level nav items that feel considered, not generic ("Dashboard" is lazy — is there something better?) 8. Onboarding tooltip sequence: 3 coach marks for first-time users. Keep each under 20 words. No "Welcome to Luma!" energy. Output as markdown with each screen as a section header.
Generate a complete UI microcopy document for [product name]'s [screen or flow]. Cover: - Page titles and headings - Primary and secondary CTA labels - Empty states: headline + 1 sentence + CTA for each major empty state - Error messages: network, validation, limits - Success and confirmation copy - Tooltip text for key features - Onboarding text and coach marks - Input placeholder text Tone: [describe voice]. Avoid: [words or patterns to exclude]. Output as structured markdown by UI section.
Why this matters

Placeholder copy causes stakeholders to react to the layout instead of the experience. Real copy shifts the entire conversation in review. Generate it first, attach it when building — the difference in feedback quality is immediate.

Step 05

User Flow

Map the journey before you build any screen.

Ask Make to generate a clickable HTML user flow — an interactive diagram of screens and decision points. Faster to produce than a Figma flow, works in any browser, and is far easier for non-designers to review asynchronously.

Figma Make Prompt
Build an interactive HTML user flow diagram for the Luma new project creation flow. Here's exactly what happens: Entry point: User clicks the "New project" button from the dashboard (top-right, primary CTA). They might also land here from the empty state CTA on first use — same flow, different entry. Step 1 — Project name Single text input, autofocused. Inline validation: if a project with this name already exists for this user, show an inline error before they move on ("You already have a project called this — want to rename it?"). Do not wait for form submission. Step 2 — Client Dropdown with search. Three paths: A. Select an existing client → move to Step 3 B. Type a name that doesn't match → offer "Create '[typed name]' as a new client" as the last option in the dropdown C. Skip entirely → "No client" option at the top of the dropdown Step 3 — Billing type Three options displayed as a radio card group: Hourly, Fixed price, Retainer. - Selecting Hourly reveals a rate input ($/hr) - Selecting Fixed reveals a total amount input - Selecting Retainer reveals a monthly amount + billing date input - Any of these can be left blank and set later — show a "Set later" affordance Step 4 — Deadline (optional) Date picker. Optional — visible "Skip for now" link. If the project has a client with other projects, suggest pulling the pattern from those (show last used deadline range as a ghost suggestion). Step 5 — Review & create Show a summary card of everything selected. Editable inline. Primary CTA: "Create project". This step only exists if at least 3 fields are filled in — if the user filled in only name and client, skip straight to creation. Error states to show: - Duplicate name (Step 1, inline) - Network failure on save (Step 5, toast) Show the happy path in one colour, skip paths in a muted variant, error paths in a third. Label every arrow. Style it as a shareable dark-background diagram. Nodes should be rounded rectangles with the step name and a one-line description inside.
Build an interactive HTML user flow diagram for [product name]'s [flow name]. Flow starts: [trigger action] Flow ends: [success state] Steps: [List key steps and decision points] Show: happy path, any skip paths, [N] error states. Style: dark background, styled nodes, directional arrows. Shareable with non-technical stakeholders.
Unexpected use case

Generate the flow before you have any opinion on it — just describe the user goal. Make will often surface steps you forgot or decision points you hadn't considered. Use it as a thinking tool first, documentation tool second.

Step 06

Build the Component

Now you're ready. And you have context behind you.

This is the prompt most people start with. Starting here without steps 1–5 means Make guesses at the problem, the user, the brand, and the copy. With those artifacts attached via the + button, this becomes a fundamentally different request.

Attach: brief · personas · brand doc · microcopy
Build Luma's project dashboard screen as a React component. I'm attaching the design brief, persona doc, brand system, and microcopy doc — use them. Don't guess at brand decisions; if the brand doc specifies something, follow it exactly. Layout I want to see: Left sidebar (fixed, 220px): - Luma wordmark at the top with a subtle version indicator - Navigation: Overview, Projects, Clients, Invoices, Time Log — each with an icon and label - Active state must be clear but not heavy - Bottom of sidebar: user avatar + name + a settings gear icon - The sidebar should feel substantial but not take over — this is where users orient, not where they work Main content area: - Greeting header: "Good morning, [First name]" — use the name "Maya" for the prototype. Below it, today's date and a one-line status ("3 projects due this week"). This should feel glanceable, not dashboard-y. - Three KPI cards in a row: Active projects (count), Pending invoice amount (£ value), Hours logged this week (number + bar showing progress toward a weekly goal). These are read-only — clicking one navigates to that section. - Project list below: show 5 projects. Each row: project name, client name, a coloured status pill (In progress / Awaiting feedback / Overdue / On hold), next deadline, and a quick-action button ("Log time" or "Send invoice" depending on status). The Overdue row should feel visually different — not alarming, just notable. - Empty state: if the project list were empty, show the copy from the attached microcopy doc (do not write new copy here). Specific requirements: - No purple. No blue. Follow the brand doc colour palette. - Use the mono font from the brand doc for all numerical values (times, amounts, counts) - Overdue status pill: use the semantic error colour from the brand doc, low opacity fill, not a bright red badge - Hover state on each project row should reveal the quick-action button — don't show it all the time - The whole component should be keyboard navigable — tab order should be logical Tech: React functional component, Tailwind for styling, no other dependencies. Single file.
Build [product name]'s [screen or component name] as a [React component / HTML page]. Using the attached design brief, personas, brand doc, and microcopy as your guide: Layout: [Describe key sections and hierarchy] Requirements: - Match brand tokens from the attached doc exactly - Use copy from the microcopy doc — no placeholder text - Optimise for [primary persona + key behaviour] - Hover states and transitions throughout Tech: [React + Tailwind / HTML + CSS]. Single file.
The compounding effect

By attaching 4 documents, you're giving Make the equivalent of a design system + user research + creative brief + content strategy in one session. The output difference compared to a blank prompt is categorical, not incremental.

→Use Website mode for full pages, Component mode for isolated UI elements.
→One screen at a time. Scope creep in the prompt produces scope creep in the output.

Step 07

Iterate & Refine

The first output is a skeleton. These prompts shape it.

Follow-up prompts are where most of the craft happens. These six patterns each target a different failure mode in generated output. Use them individually — one at a time — and always describe what you want rather than what went wrong.

High-Leverage Follow-Up Prompts
// Hierarchy problem — the KPI cards are competing with the project list "Right now my eye goes to the three KPI cards first, but they're read-only information — the project list is where I actually need to take action. Reduce the visual weight of the KPI cards by about 40%: smaller type, less contrast, lighter borders. The project list rows should feel like the primary content. Don't move anything — just adjust hierarchy." // Accessibility — focus states are invisible "Tab through the entire component and fix everything you find: any interactive element without a visible focus ring, any colour combination that fails WCAG AA contrast (check the status pills especially — the Overdue pill is probably failing), and the project rows should be focusable as a unit with keyboard Enter triggering the quick action." // Responsive — sidebar breaks below 900px "At viewport widths below 900px: collapse the sidebar to 64px (icons only, no labels), with a toggle button to expand it to full width as an overlay. At below 600px: move navigation to a bottom tab bar with 5 icons (hide labels). The KPI cards should stack to a single column below 600px. The project list stays full width throughout." // The greeting header feels generic — fix the personalisation "The 'Good morning, Maya' header is working but the status line underneath ('3 projects due this week') is too vague to be useful. Replace it with something context-aware: if there's a project overdue, surface that specifically. If the nearest deadline is today, say that. If everything is on track, say something that actually reflects that. Write 4 different versions of this line for different states and implement the logic to show the right one." // Data states — the component only handles the happy path "Add three missing states to the project list: (1) Loading — skeleton rows using the same dimensions as the real rows, animated shimmer effect, show 5 of them. (2) Empty — use the copy from the attached microcopy doc exactly as written. (3) Error — a network request failed. Show an inline error message with a retry button. All three should match the existing brand exactly." // Persona check — run it against the overwhelmed persona "Read the second persona in the attached personas doc — the one who uses spreadsheets and describes themselves as 'organised in my own chaotic way'. She's opened Luma for the first time and is looking at this dashboard. Based on what you know about her, tell me: what are the 3 things most likely to confuse or frustrate her in the first 30 seconds? Then fix those 3 things without breaking what's already working for the other personas."
// Hierarchy Fix "The most important action is [action]. Redesign visual hierarchy so it draws the eye first." // Accessibility "Run an accessibility review. Fix contrast, focus states, missing labels, and form field associations." // Responsive "Update for mobile: [describe breakpoint behaviour]." // Motion "Add micro-interactions: [hover states, transitions, load animations]." // Data States "Add: loading skeleton, empty state with copy '[microcopy]', error state. All on-brand." // Persona Check "Review against the [persona name] persona in the attached doc. What frustrates them? Fix the top 3 issues."
The Persona Check

The last prompt is the most underused. Asking Make to critique its own output through the eyes of a specific user — with the persona doc attached — surfaces issues that no amount of visual polish will fix. Run it before you call any screen done.

→One change at a time. Stacking instructions leads to unpredictable changes across the component.
→If a change goes wrong, describe what you wanted rather than what you got.
→Save versions you like before making significant changes — Make doesn't have granular undo.
